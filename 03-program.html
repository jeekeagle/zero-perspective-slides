把提示词当作源码,看它如何获得类型、模块、版本
这些程序性特征 —— 一种运行在大模型上的 DSL。

Zero · 2026 Vol.I · Lenses 10 Slides
Evolution · 演化

从恳切的信 → DSL

早期写提示词,和写一段"恳切的信"差不多:礼貌、详细、希望对方能懂。

这种提示词就像一份没有注释、没有类型、全部塞在一个文件里的 PHP 早期代码 —— 能跑,但没人能维护。

今天的情况变了。一个严肃的 Prompt 工程,正在向一种新的领域特定语言(DSL)演化:

System   / Developer / User      ← 结构
{ topic } / { tone } / { date }   ← 变量
@tool / @skill / sub-prompt       ← 函数
import './shared.md'              ← 模块
v1.2.3 (Git)                      ← 版本
golden set / eval suite           ← 测试
Definition · 定义

你几乎可以把它当作一种
"运行在大模型上的
脚本语言"。

—— 像 Python 之于 CPU,Prompt 之于 LLM。

Anatomy · 程序特征

一段 prompt 拆开看

// 静态结构

  • 类型:JSON schema / enum / format
  • 模块:import 别人的 prompt 或 tool
  • 版本:v1.2.3(Git 仓库)

// 运行时

  • 函数:@tool / @skill / sub-prompt
  • 测试:eval suite / golden set
  • 类型检查:JSON schema + validator

6 个特征 = 6 种"程序性"。一段 prompt,从此不再是一段散文。

Runtime · 提示词的"运行时"

一段 prompt 不是孤立执行的

用户输入 → 模板渲染 → 上下文组装 → 模型调用
↑ ← 工具调用 / 子 prompt → 结果解析 + 校验
↺ 失败重试 → 输出

看清这条链路,你就理解了为什么"调 prompt"常常不是改文本,而是在改这条链路里的某一段。

Version · 被严重低估的环节

版本管理

  • 01
    v1 与 v2 可能差异巨大
    模型换了 / 上下文窗口变了 / 加了示例 —— 行为可能差到不像同一个东西。
  • 02
    文件名后缀不是版本管理
    很多人还在用 prompt-v3-final.md 管它 —— 这不是工程,是仪式。
  • 03
    没有版本管理 = 永远的会
    迟早让团队"以谁的版本为准"开一场永远开不完的会。
!
Prompt 像程序,
但不是程序。

— 必须诚实地划这条线

  • ×没有严格的执行顺序
  • ×没有形式化的语义
  • ×没有确定的输出

一种"近似确定性"的程序。linter / formal verification 全失效 —— 需要新工具。
