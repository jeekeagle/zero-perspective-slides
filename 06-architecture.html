The Lens of Architecture · Deck 06 of 06

从知识的版本、引用、按需加载,
看 RAG 改变了什么、没改变什么。

Zero · 2026 Vol.I · Lenses 10 Slides
Problem · 一个朴素的问题

模型"知道"的事是有限的

它的训练数据截止那一刻为止。

而且它在那些事上的记忆也是压缩的、扭曲的 —— 参数量是有限的。

一个朴素的问题:模型不知道的东西越来越多,记错的东西也在增长。

RAG 的回应很直接: 别让模型什么都记,让它在需要的时候去查。

Definition · 定义

RAG 把"知识"
从模型参数里
搬到了外部存储里。

模型的注意力被"按需支付"在真正相关的知识上 ——
而不是把所有可能性塞进上下文窗口。

Three States · 知识的三种状态

RAG 让知识有了第三种状态

// S1 · 参数化记忆

在模型权重里

快 · 模糊 · 难改

// S2 · 上下文窗口

在当前 prompt 里

精确 · 有上限 · 易失

// S3 · 外部知识库 ← NEW

在 RAG 后端

无限 · 可改 · 可治理

// 这是范式变化

原来只有前两种;现在有了第三种。

S3 会越来越主导。

"
可追溯,等于可证伪。
可证伪,等于可信任。

— Citation · RAG 最大的隐性收益

但这是有代价的:你的知识库必须可治理 ——
来源要可信 / 版本要可追 / 访问要可控。

一份被污染的知识库,会让模型系统性地胡说八道。

Pipeline · 按需加载的工程

RAG 是按需加载的极致实现

用户问 → 改写 query → 向量检索 → 重排
↳ top-k 文档 → 拼进 prompt → 模型生成

改写 · embedding · chunking · reranking · 压缩 · citation

每一步都是一个独立的优化空间,都有自己的坑和论文。

Limit · RAG 没改变什么

实话时间

  • ×
    RAG 没改变模型本身的智能上限
    该胡说的地方,RAG 也救不了;本来就不会的推理,RAG 也教不会。
  • ×
    它改变的是 表现,不是 智能
    在特定任务上的表现、对新鲜事物的响应速度、可治理性 —— 这些是"工程层面"的改进。
  • ×
    RAG 是 脚手架,不是 地基
    地基还是模型的预训练 + 推理能力。脚手架再好,墙还是会倒。
3-Layer Stack · 三层架构

未来三年的标准 stack

L1
Model
// 通用智能
负责"会做" —— 推理、生成、对话
L2
RAG
// 事实与新鲜度
负责"对的" —— 引用、可追溯、可治理
L3
Workflow
// 稳定可重复
负责"稳定" —— Agent / Skill / SOP
stack
// 合在一起
当一个 stack 来设计 / 运维 / 治理
Judgment · 一个判断

把这三层当一个 stack来设计、来运维、来治理,
会变成Agent 时代最基本的工程素养。

缺任何一层,系统都会在某个时刻崩。
顺序错,会更早崩。

Model + RAG + Workflow = The Stack
